# proxy-browser

Проект расширения для Firefox, которое позволяет выбирать сайты и связанные с ними домены, которые должны ходить через указанный прокси.

## Цель v1

Первая версия делается под Firefox и ориентируется на задачу:

- скрыть трафик от провайдера;
- скрыть трафик от локальной сети;
- проксировать только выбранные сайты и связанные с ними запросы, а не весь браузер целиком.

## Почему базовый вариант это HTTPS proxy

Для текущей задачи основным типом прокси выбираем `HTTPS proxy`.

Причина:

- соединение между Firefox и прокси шифруется;
- провайдер и локальная сеть видят подключение к прокси-серверу, но получают меньше информации о конечных сайтах, чем при обычном `HTTP proxy` или `SOCKS5`;
- это лучше соответствует задаче "скрыть трафик от провайдера или локальной сети".

## Поддержка прокси

В проекте нужно закладывать поддержку нескольких типов прокси Firefox, но основной сценарий для v1 такой:

- основной тип: `HTTPS proxy`;
- дополнительные типы на будущее: `HTTP`, `SOCKS5`, `SOCKS4`;
- `Xray/VLESS/VMESS/Trojan/Shadowsocks` напрямую расширением не поддерживаются;
- такие системы можно использовать только через внешний локальный клиент, который поднимает обычный локальный proxy endpoint, например `HTTPS` или `SOCKS5`.

## Поведение расширения

Планируемая механика работы:

1. Пользователь открывает нужный сайт.
2. Открывает popup расширения.
3. Нажимает кнопку добавления сайта в проксирование.
4. Расширение запоминает хост из адресной строки как правило для сайта.
5. Расширение собирает сетевые хосты, замеченные у этой вкладки, только как подсказку и контекст.
6. Пока вкладка открыта на этом сайте, её запросы идут через выбранный прокси.

## Что должно уметь расширение

- хранить список прокси-конфигураций;
- хранить список сайтов, которые должны идти через прокси;
- назначать конкретный прокси конкретному сайту или группе сайтов;
- включать и выключать правило без удаления конфигурации;
- давать простой интерфейс: "Добавить в proxy" для текущей страницы.

## Почему правило лучше привязывать к сайту, а не к общим доменам

У разных сайтов часто бывают одинаковые внешние зависимости:

- общие CDN;
- аналитика;
- шрифты;
- API на общих доменах;
- медиа-ресурсы.

Если проксировать только по доменам ресурсов, можно получить побочный эффект:

- домен `cdn.example.net` нужен для сайта `site-a.com` через proxy;
- этот же домен используется на `site-b.com`, где proxy не нужен;
- тогда простое доменное правило будет ошибочно проксировать оба сайта.

Поэтому базовая модель для v1 должна быть такой:

- правило определяется по хосту в адресной строке;
- proxy включается для сетевого контекста этой вкладки;
- собранные домены используются как подсказка и отладочная информация, а не как глобальное правило маршрутизации.

## Ограничения и честные ожидания

Важно явно зафиксировать ограничения:

- расширение управляет только трафиком Firefox;
- прокси не делает трафик полностью невидимым, а только меняет маршрут и уровень видимости метаданных;
- сам прокси-сервер остаётся доверенной точкой и видит назначения;
- если нужен более сильный уровень маскировки, нужен внешний клиент вроде Xray, sing-box или Clash, а расширение должно работать уже через их локальный `HTTPS` или `SOCKS5` endpoint.

## Направление реализации

Текущее направление проекта:

- браузер: `Firefox`;
- приоритет по приватности: скрыть трафик от провайдера и локальной сети;
- основной протокол для v1: `HTTPS proxy`;
- архитектура: расширение собирает домены страницы и маршрутизирует только их через выбранный прокси.

## Текущее состояние прототипа

Сейчас в проекте уже заложен базовый каркас расширения:

- `manifest.json` для Firefox WebExtensions;
- фоновый скрипт, который хранит конфигурацию и применяет proxy-правила;
- popup-интерфейс с тремя вкладками: `Главная`, `Настройка`, `Прокси`;
- отдельный сценарий первого запуска без сохранённого proxy;
- кнопка добавления текущего сайта в proxy-правила;
- сбор замеченных хостов активной вкладки через `webRequest`.

В текущем виде прототип:

- поддерживает несколько `HTTPS proxy` и `HTTP proxy` профилей;
- хранит их локально у клиента;
- поддерживает глобальное состояние `Вкл / Выкл`;
- поддерживает proxy по умолчанию;
- создаёт правило по хосту сайта в адресной строке;
- позволяет для конкретного сайта назначить отдельный proxy через меню в карточке;
- применяет proxy к запросам вкладки, если текущий сайт совпадает с сохранённым правилом и расширение включено;
- показывает замеченные по вкладке хосты как подсказку для анализа страницы.

## Как запустить локально

1. Открыть Firefox.
2. Перейти на `about:debugging`.
3. Открыть раздел `This Firefox`.
4. Нажать `Load Temporary Add-on`.
5. Выбрать файл `manifest.json` из папки проекта.

После этого можно:

1. Открыть нужный сайт.
2. Открыть popup расширения.
3. Если proxy ещё нет, на вкладке `Главная` нажать `Добавить proxy`.
4. На вкладке `Прокси` сохранить proxy-профиль.
5. При необходимости сделать его proxy по умолчанию.
6. Вернуться на вкладку `Главная` и включить `Вкл`.
7. Добавить текущий сайт в proxy.

## Ближайшие улучшения

- поддержка `SOCKS5` и `SOCKS4`;
- импорт нескольких proxy-конфигураций;
- более аккуратная фильтрация служебных и сторонних доменов;
- редактирование уже сохранённых proxy-профилей;
- отдельное быстрое назначение proxy для текущего сайта прямо на главной вкладке;
- более явная индикация того, какой proxy реально выбран для текущей вкладки.
